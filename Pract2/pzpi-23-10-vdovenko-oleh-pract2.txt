   3. ОПИС ВИКОНАНОЇ РОБОТИ
3.1. Загальні відомості про систему
     Назва системи: Twitter (з 2023 року функціонує під брендом X). Розробник: Twitter Inc. (на поточний момент інтегрована у структуру X Corp.).
     Призначення системи: Платформа Twitter (X) є глобальною системою мікроблогінгу та соціальною мережею, що орієнтована на обмін інформацією, мультимедійним контентом та новинами у режимі реального часу (Real-time Web).
     Область застосування: Система застосовується у сферах масових комунікацій, цифрового маркетингу, агрегації новин, швидкого реагування на надзвичайні події, а також виступає потужним інструментом для аналітики великих даних (Big Data) та збору розвідувальної інформації на основі відкритих джерел (OSINT). Платформа забезпечує інфраструктуру як для B2C (кінцеві користувачі), так і для B2B сегментів (надання доступу до потокових даних через API для сторонніх розробників та аналітичних агентств).
     Користувачі системи: Аудиторія платформи є глобальною та багатофункціональною. Станом на поточний момент система обслуговує понад 500 мільйонів активних користувачів щомісяця (MAU). До основних категорій користувачів належать: звичайні споживачі контенту, автори (блогер-інфлюенсери), корпоративні акаунти, урядові та державні установи, засоби масової інформації, а також автоматизовані системи (боти), що взаємодіють з платформою через програмні інтерфейси.
     Основні функції та технічні вимоги (SLA):
• Публікація та розповсюдження контенту (Broadcasting): Створення коротких повідомлень (твітів) із вкладенням медіафайлів. Головний інженерний виклик — забезпечення миттєвої доставки повідомлення всім підписникам.
• Керування соціальним графом: Формування та зберігання зв'язків між мільйонами вузлів (підписки, читачі, блокування), що вимагає специфічних графових баз даних для швидкого обчислення перетинів.
• Генерація персоналізованих стрічок (Timelines): Об'єднання публікацій від підписок користувача в єдиний хронологічний або алгоритмічний потік із затримкою (latency), що не повинна перевищувати 200 мілісекунд.
• Повнотекстовий пошук у реальному часі: Індексація нових твітів та забезпечення їх доступності у глобальному пошуку вже через 3-5 секунд після публікації.
• Аналіз трендів (Trending Topics): Безперервна потокова обробка вхідних даних для виявлення популярних тем та хештегів у різних географічних регіонах.
• Стійкість до екстремальних навантажень: У штатному режимі система обробляє близько 6 000 запитів на запис (Writes) та понад 300 000 запитів на читання (Reads) щосекунди. Проте архітектура спроєктована з розрахунком на пікові сплески (наприклад, під час спортивних фіналів чи виборів), витримуючи понад 150 000 публікацій щосекунди без деградації основного функціонала.
3.2. Архітектурний стиль системи
     Визначення архітектурного стилю: Сучасна платформа Twitter (X) побудована на базі розподіленої мікросервісної архітектури (Microservices Architecture), яка тісно інтегрована з принципами подієво-орієнтованої архітектури (Event-Driven Architecture). Цей гібридний підхід дозволяє системі поєднувати незалежність окремих вузлів із можливістю асинхронної реакції на мільйони подій (публікацій, лайків, ретвітів) у реальному часі.
     Історичний контекст та проблематика моноліту: Для повного розуміння поточного архітектурного стилю необхідно розглянути еволюцію системи. У перші роки свого існування (2006–2010) Twitter був класичним монолітним веб-додатком, який розробники неофіційно називали «Monorail». Він був написаний з використанням фреймворку Ruby on Rails та спирався на єдину реляційну базу даних MySQL.
     Такий стиль дозволив стартапу швидко вийти на ринок, проте з експоненційним зростанням аудиторії моноліт продемонстрував свої фундаментальні обмеження. Синхронна обробка запитів та глобальне блокування інтерпретатора (GIL) у Ruby призводили до того, що сервери витрачали більшість часу на очікування відповідей від бази даних (I/O блокування). База даних, у свою чергу, страждала від взаємних блокувань таблиць (deadlocks) під час одночасних спроб запису десятків тисяч нових твітів. Наслідком цього стали постійні каскадні відмови всієї системи, що супроводжувалися появою відомої сторінки помилки «Fail Whale».
     Обґрунтування переходу до мікросервісної архітектури: Критичні проблеми з продуктивністю змусили інженерів Twitter ухвалити рішення про радикальний рефакторинг та перехід до сервіс-орієнтованої (SOA), а згодом — мікросервісної архітектури. Обґрунтування цього вибору базується на наступних ключових перевагах:
1. Незалежне горизонтальне масштабування (Independent Scalability): У моноліті всі функції (реєстрація, публікація, пошук) масштабуються разом, що є неефективним використанням апаратних ресурсів. Мікросервісний підхід дозволив Twitter масштабувати лише ті компоненти, які перебувають під найбільшим навантаженням. Наприклад, сервіс генерації стрічки новин (Timeline Service) можна розгорнути на тисячах серверів, тоді як сервіс редагування профілю працюватиме лише на десятках.
2. Ізоляція збоїв та плавна деградація (Fault Isolation & Graceful Degradation): Якщо в моноліті виникає критична помилка в модулі (наприклад, витік пам'яті в підсистемі рекомендацій), «падає» весь сервер. У мікросервісній архітектурі збій одного сервісу локалізується. Якщо сервіс підрахунку лайків перестане відповідати, користувачі все одно зможуть читати та публікувати твіти (плавна деградація функціоналу).
3. Технологічна гетерогенність (Polyglot Environment): Розподіл на мікросервіси дозволив відмовитися від єдиної технології. Компанія перейшла на платформу JVM (Java Virtual Machine), використовуючи мови Scala та Java. Scala була обрана завдяки її функціональним парадигмам, моделі акторів та потужним інструментам для асинхронного, неблокуючого програмування (зокрема, фреймворку Netty), що ідеально підходять для HighLoad систем. Крім того, кожен мікросервіс отримав можливість використовувати ту базу даних, яка найкраще підходить для його специфічного завдання (Polyglot Persistence).
4. Організаційна гнучкість та CI/CD: Монолітна архітектура вимагала синхронізації роботи сотень розробників для одного релізу. Мікросервіси дозволили розділити інженерний відділ на невеликі автономні команди. Кожна команда стала власником свого сервісу, отримавши можливість незалежно розробляти, тестувати та розгортати код у продакшн по кілька разів на день без ризику зламати роботу інших відділів.
3.3. Основні компоненти системи
     Архітектура платформи Twitter (X) є яскравим прикладом багаторівневої топології, де кожен компонент виконує суворо визначену функцію та ізольований від інших для забезпечення максимальної відмовостійкості. Нижче наведено детальний опис основних підсистем платформи.
     Інтерфейс користувача (Клієнтський рівень) Клієнтський рівень представлений гетерогенною екосистемою додатків, що взаємодіють із серверною частиною через стандартизовані API (REST та GraphQL).
• Вебклієнт: Побудований як прогресивний вебдодаток (PWA) з використанням сучасних JavaScript-фреймворків (зокрема React). Для оптимізації пошукової видачі (SEO) та зменшення часу до першого відмальовування (Time to First Byte) активно використовується серверний рендеринг (SSR).
• Мобільні клієнти: Нативні додатки для операційних систем iOS та Android, оптимізовані для роботи в умовах нестабільного мережевого з'єднання. Вони реалізують складні механізми локального кешування (наприклад, за допомогою SQLite) для забезпечення офлайн-доступу до раніше завантажених стрічок новин.
     Серверна частина (Межовий рівень / Edge Layer) Цей рівень є єдиною точкою входу для всіх клієнтських запитів до системи, виконуючи роль надійного бар'єра між зовнішнім світом та внутрішніми мікросервісами.
• API Gateway та Балансувальники навантаження: Система використовує апаратні та програмні балансувальники для рівномірного розподілу вхідного трафіку між дата-центрами. Межові сервери здійснюють термінацію SSL/TLS-з'єднань, що знімає криптографічне навантаження з внутрішніх мікросервісів.
• Фільтрація та безпека: На цьому етапі реалізовані механізми обмеження частоти запитів (Rate Limiting) для захисту від DDoS-атак та парсингу даних, а також відбувається первинна аутентифікація запитів (перевірка OAuth-токенів) перед їхньою маршрутизацією у внутрішню мережу.
     Рівень бізнес-логіки (Microservices Layer) Це «ядро» платформи, яке складається з сотень мікросервісів, написаних переважно на Scala та Java. Кожен сервіс інкапсулює вузьку доменну логіку:
• Tweet Ingestion Service (Сервіс обробки публікацій): Приймає нові твіти від користувачів, проводить валідацію контенту, вилучає метадані (хештеги, згадки) та ініціює асинхронний процес розсилки повідомлення (Fan-out).
• Timeline Service (Сервіс формування стрічки): Найбільш ресурсомісткий компонент. Відповідає за об'єднання твітів від підписок користувача у хронологічну або алгоритмічну стрічку. Цей сервіс не звертається безпосередньо до баз даних при кожному запиті клієнта, а працює з попередньо сформованими матеріалізованими представленнями в оперативній пам'яті.
• Social Graph Service (Сервіс соціального графа): Управляє всіма зв'язками між користувачами (підписки, читачі, блокування, мути). З огляду на те, що архітектура Twitter базується на асиметричному графі (користувач може читати іншого без взаємності), цей сервіс оптимізовано для надшвидкого обчислення перетинів великих множин.
     Рівень зберігання даних (Storage Layer) Компанія застосовує парадигму Polyglot Persistence, підбираючи тип сховища під конкретні вимоги до швидкості та консистентності:
• Розподілені реляційні бази (Gizzard/MySQL): Використовуються для збереження критично важливої транзакційної інформації (дані облікових записів, налаштування безпеки). Фреймворк Gizzard автоматизує процес глибокого хеш-шардування (поділу даних на дрібні фрагменти), що дозволяє MySQL масштабуватися горизонтально на тисячі вузлів.
• Розподілені NoSQL сховища (Manhattan): Власна розробка компанії, оптимізована для низьких затримок (low latency) при читанні та записі. Використовується як основне сховище (Source of Truth) для самих текстів твітів, прямих повідомлень (Direct Messages) та метаданих. Manhattan підтримує консистентність у кінцевому рахунку (Eventual Consistency) та автоматичну реплікацію між континентами.
• Сховища мультимедіа (Blob Storage): Відокремлена інфраструктура для збереження петабайтів фото- та відеоконтенту. Тісно інтегрована з глобальними мережами доставки контенту (CDN), щоб користувач отримував «важкі» файли з найближчого до нього географічного сервера.
     Допоміжні сервіси та кешування Ці компоненти критично важливі для забезпечення роботи системи в режимі реального часу:
• Системи In-Memory кешування (Redis та Twemcache): Відіграють ключову роль у продуктивності платформи. Сотні терабайтів оперативної пам'яті використовуються для зберігання готових стрічок новин (Home Timelines) активних користувачів. Читання відбувається безпосередньо з RAM, що гарантує час відповіді у кілька мілісекунд.
• Сервіс ідентифікації (Snowflake): Інфраструктурний компонент, що вирішує проблему генерації унікальних ID у розподіленій системі. Snowflake децентралізовано створює 64-бітні ідентифікатори на основі часових міток без звернення до бази даних і без блокувань.
• Пошуковий індекс (Earlybird): Спеціалізована система на базі технології Apache Lucene. Здійснює індексацію вхідного потоку твітів просто в оперативній пам'яті, що дозволяє результатам з'являтися у глобальному пошуку за ключовими словами менш ніж через 5 секунд після публікації.
3.4. Взаємодія компонентів
     Взаємодія між сотнями мікросервісів у системі Twitter має складну гібридну природу. Система використовує різні підходи до маршрутизації потоків даних, застосовуючи як швидкі синхронні протоколи, так і надійні асинхронні механізми синхронізації.
     Протоколи взаємодії та потоки даних: На зовнішньому рівні комунікація між клієнтськими додатками та межовим рівнем (API Gateway) відбувається за допомогою стандартних протоколів HTTPS (REST API та GraphQL). Однак для внутрішньої взаємодії між мікросервісами використання HTTP є занадто повільним. Тому компанія використовує протокол RPC (Remote Procedure Call) на базі власного фреймворку Finagle. Дані передаються у бінарному форматі за допомогою протоколу серіалізації Apache Thrift, що радикально зменшує мережевий трафік та пришвидшує потоки даних.
     
     Опис діаграми: Діаграма компонентів ілюструє статичну структуру системи. Чітко простежується перехід від текстових протоколів (REST) на межовому рівні до високоефективних бінарних протоколів (Finagle/Thrift) всередині кластера. Apache Kafka виступає центральною ланкою для асинхронних потоків даних між мікросервісами.
     Механізми транзакцій та синхронної обробки: Коли клієнт ініціює операцію (наприклад, публікацію твіта), потік даних розділяється. Перша частина процесу є синхронною: система має згенерувати ID, зберегти первинний запис у NoSQL базу Manhattan та відповісти клієнту. Це забезпечує консистентність даних з точки зору автора публікації. Лише після успішної синхронної транзакції ініціюється фоновий процес синхронізації кешів.
     Опис діаграми: Діаграма послідовності демонструє тимчасові рамки потоку даних. Основний акцент зроблено на механізмі "раннього підтвердження": мікросервіс завершує синхронну транзакцію з базою даних і відразу відправляє успішну відповідь користувачу, не чекаючи завершення важких процесів розсилки контенту підписникам.
     Механізми синхронізації та проблема Fan-out: Найскладнішим механізмом синхронізації у системі є процес оновлення стрічок новин (Fan-out). Платформа застосовує гібридну стратегію синхронізації даних, яка залежить від соціального графа користувача. Для звичайних користувачів використовується механізм асинхронної фонової синхронізації (Push), а для знаменитостей — механізм синхронізації за запитом (Pull).

     Опис діаграми: Діаграма діяльностей розкриває алгоритмічний механізм розгортання графа підписок. Відображено логічне розгалуження потоку виконання: система динамічно обирає між ресурсомісткою асинхронною розсилкою (Push) та економною синхронізацією під час читання (Pull), тим самим захищаючи кластери кешування від перевантаження.
3.5. Масштабованість та надійність
     Архітектура платформи проєктувалася з урахуванням вимог до надвисокої доступності (High Availability) та здатності витримувати раптові піки навантаження, що унеможливлюють використання стандартних підходів до масштабування.
     Механізми балансування навантаження: Балансування реалізовано на кількох рівнях архітектури. На зовнішньому межовому рівні (Edge) використовуються апаратні балансувальники навантаження та системи Anycast DNS, які маршрутизують запити клієнтів до найближчого за географічним розташуванням дата-центру. На внутрішньому рівні між мікросервісами застосовується клієнтське балансування навантаження (Client-Side Load Balancing), реалізоване через фреймворк Finagle. Замість класичного алгоритму Round-Robin, Finagle використовує алгоритми на основі експоненційно згладженого рухомого середнього (EWMA), що дозволяє динамічно визначати найменш завантажені вузли та спрямовувати трафік до них, мінімізуючи загальну затримку (latency).
     Відмовостійкість: Для запобігання каскадним відмовам у системі реалізовано патерн Circuit Breaker (Запобіжник). Якщо певний мікросервіс починає відповідати з помилками або перевищує таймаути, комунікація з ним тимчасово блокується, щоб дати йому час на відновлення. Крім того, платформа широко використовує принцип плавної деградації (Graceful Degradation): у разі падіння допоміжних систем (наприклад, сервісу рекомендацій або аналітики), базовий функціонал (читання стрічки та публікація нових твітів) продовжує працювати безперебійно.
     Кешування: Агресивне кешування є основою швидкодії системи. Для зберігання попередньо обчислених стрічок новин (Timelines) активних користувачів використовуються гігантські кластери In-Memory сховищ на базі Redis та власної модифікації Memcached (Twemcache). Читання стрічки відбувається виключно з оперативної пам'яті, що гарантує час відповіді у кілька мілісекунд (O(1)). Кеш інвалідується або оновлюється асинхронно механізмами Fan-out при появі нових публікацій.
     Реплікація: Для забезпечення надійності та консистентності даних система використовує багаторівневу реплікацію. Реляційні бази даних MySQL реплікуються за схемою Master-Slave з автоматичним перемиканням ролей у разі збою (Failover). NoSQL-база Manhattan використовує кворумну реплікацію, забезпечуючи розповсюдження даних між кількома ізольованими дата-центрами. Це гарантує, що навіть повне фізичне знищення або відключення одного дата-центру не призведе до втрати даних користувачів.
     Контейнеризація або хмарна інфраструктура: Для управління тисячами мікросервісів та ефективної утилізації апаратних ресурсів компанія застосовує сучасні методи оркестрації. Історично платформа використовувала власні рішення на базі Apache Mesos та Aurora, які дозволяли автоматизувати розгортання та перезапуск сервісів. Сучасна інфраструктура тяжіє до використання стандартизованих контейнерних технологій (Docker) та міграції в середовище Kubernetes. Це забезпечує декларативне управління інфраструктурою (Infrastructure as Code), автоматичне горизонтальне масштабування (Auto-scaling) при сплесках трафіку та ізоляцію процесів на рівні операційної системи.
3.6. Приклади програмної реалізації
     Для демонстрації архітектурних принципів високонавантажених розподілених систем у роботі наведено програмну реалізацію алгоритму Twitter Snowflake мовою C# (повний лістинг коду та запити до ШІ наведено у Додатку В).
     Демонстрація архітектурного принципу: У мікросервісній архітектурі використання традиційних баз даних для генерації первинних ключів (наприклад, через AUTO_INCREMENT) створює єдину точку відмови та «вузьке місце» (bottleneck) для продуктивності. Представлений програмний код демонструє принцип децентралізованої генерації ідентифікаторів. Алгоритм дозволяє кожному серверу незалежно генерувати унікальні 64-бітні ID без мережевої взаємодії з іншими вузлами, що забезпечує нульову затримку.
     Опис фрагмента коду: Програмна реалізація базується на використанні побітових операцій (побітовий зсув вліво << та логічне АБО |). Код формує єдине 64-бітне число (тип long) з трьох компонентів:
1. Часова мітка (Timestamp) — займає 41 біт і забезпечує природне хронологічне сортування записів.
2. Ідентифікатор вузла (Worker ID) — займає 10 біт і гарантує відсутність колізій між різними мікросервісами.
3. Порядковий номер (Sequence) — займає 12 біт і дозволяє одному вузлу генерувати до 4096 унікальних ідентифікаторів протягом однієї мілісекунди. Код реалізовано з використанням конструкції lock для забезпечення потокобезпечності (Thread Safety) при паралельних запитах на одному сервері.
     Опис запитів до сервісів ШІ: У процесі виконання практичної роботи інструменти генеративного штучного інтелекту використовувалися як допоміжний засіб для архітектурного аналізу. Основні напрямки використання:
• Збір та синтез інформації щодо еволюції платформи від моноліту до SOA.
• Написання оптимізованого еталонного коду алгоритму Snowflake мовою C# з урахуванням обробки крайових випадків (наприклад, переведення системного годинника назад).

1 using System;
2 
3 public class SnowflakeIdGenerator
4 {
5     private const long Twepoch = 1288834974657L; // Епоха Twitter (4 листопада 2010 року)
6     private const int WorkerIdBits = 10;
7     private const int SequenceBits = 12;
8     
9     private const long MaxWorkerId = -1L ^ (-1L << WorkerIdBits);
10     private const long SequenceMask = -1L ^ (-1L << SequenceBits);
11     
12     private const int WorkerIdShift = SequenceBits;
13     private const int TimestampLeftShift = SequenceBits + WorkerIdBits;
14 
15     private readonly long _workerId;
16     private long _sequence = 0L;
17     private long _lastTimestamp = -1L;
18     private readonly object _lock = new object();
19 
20     public SnowflakeIdGenerator(long workerId)
21     {
22         if (workerId > MaxWorkerId || workerId < 0)
23         {
24             throw new ArgumentException($"Worker ID повинен бути в діапазоні від 0 до {MaxWorkerId}");
25         }
26         _workerId = workerId;
27     }
28 
29     public long NextId()
30     {
31         lock (_lock)
32         {
33             long timestamp = GetCurrentTimestamp();
34 
35             if (timestamp < _lastTimestamp)
36             {
37                 throw new Exception("Системний час перемістився назад. Генерація неможлива.");
38             }
39 
40             if (_lastTimestamp == timestamp)
41             {
42                 _sequence = (_sequence + 1) & SequenceMask;
43                 if (_sequence == 0)
44                 {
45                     timestamp = WaitNextMillis(_lastTimestamp);
46                 }
47             }
48             else
49             {
50                 _sequence = 0L;
51             }
52 
53             _lastTimestamp = timestamp;
54 
55             // Формування 64-бітного ID за допомогою побітових зсувів
56             return ((timestamp - Twepoch) << TimestampLeftShift) |
57                    (_workerId << WorkerIdShift) |
58                    _sequence;
59         }
60     }
61 
62     private long GetCurrentTimestamp()
63     {
64         return DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
65     }
66 
67     private long WaitNextMillis(long lastTimestamp)
68     {
69         long timestamp = GetCurrentTimestamp();
70         while (timestamp <= lastTimestamp)
71         {
72             timestamp = GetCurrentTimestamp();
73         }
74         return timestamp;
75     }
76 }
